### Assistant (2025-10-19 12:23:46)

Good question — here’s the short version first:  

> **You had to install Plex Media Server with `snap` instead of `apt` because Plex no longer maintains an up-to-date `.deb` package in the official Ubuntu repositories, and Snap is now the *officially supported distribution channel* for Plex on Ubuntu.**

Let’s break it down:  

---

### 🧩 1. **Apt vs Snap Packaging**
- `apt` installs software from **Debian/Ubuntu repositories** (traditional `.deb` packages).  
- `snap` installs **containerized applications** maintained independently from the OS repositories.  
- Snap packages are sandboxed, self-contained, and auto-updated — which Plex likes because it reduces dependency and versioning issues.

---

### 📦 2. **Why Plex Chose Snap**
- Plex wanted **one consistent build** across all Ubuntu-based distributions (18.04 → 24.04 and beyond).  
- Snap lets them package the **latest server version** without waiting for Ubuntu maintainers to update the `apt` repo.
- It simplifies maintenance for Plex developers — only one build pipeline to maintain.

---

### 🔐 3. **Permissions and Isolation**
- Snap provides controlled filesystem and network access (via confinement profiles).
- This makes Plex less likely to break due to system library mismatches or dependency upgrades.

---

### ⚠️ 4. **Why `apt install plexmediaserver` Fails**
- Ubuntu’s official repositories **don’t include Plex anymore**.
- Even if you add `deb https://downloads.plex.tv/repo/deb public main`, that repo is **often stale** or missing builds for newer Ubuntu versions (e.g., 24.04).
- So, `apt` may not find the package or will install an **older unsupported version**.

---

### 🧰 5. **What You Can Still Do (If You Prefer Apt-Style Control)**
If you dislike Snap, you *can* still install Plex manually:
```bash
wget https://downloads.plex.tv/plex-media-server-new/1.x.x.x-xxxxxxx/plexmediaserver_1.x.x.x-xxxxxxx_amd64.deb
sudo dpkg -i plexmediaserver_*.deb
```
…but you’ll need to manually update Plex when new versions come out, and dependencies may occasionally break between Ubuntu releases.

does the fact that plexmediaserver is running off of snap instead of systemd create a problem for interacting with other services such as perhaps others that are dockerized?

### Assistant (2025-10-19 12:25:54)

Short answer: **No—running Plex via Snap doesn’t inherently break integration with Dockerized services.** Snap apps are still managed by systemd (through `snapd`), expose normal TCP/UDP ports, and talk over HTTP just like a `.deb` install. Most “gotchas” people hit are about **file access paths, permissions, and Snap confinement**, not systemd.

Here’s what to watch:

### 1) Service management
- Snap services are still systemd units under the hood. You can check them with:
  ```bash
  systemctl status snap.plexmediaserver.plexmediaserver.service
  ```
  So there’s no fundamental “Snap vs systemd” conflict.

### 2) Networking between Plex and Docker apps (Sonarr/Radarr/etc.)
- **API calls:** Docker apps call Plex on `http://<host-ip>:32400` (or hostname). That works fine with the Snap.
- **Ports:** Ensure 32400/TCP (and any DLNA/remote ports you use) aren’t blocked by a firewall. No special Snap tweak needed.
- **Discovery:** If you rely on DLNA/UPnP/mDNS discovery across containers/host, keep Docker on the default bridge or use host networking as appropriate; the Snap doesn’t block normal multicast in typical setups.

### 3) Filesystem visibility (most common pitfall)
Snap confinement can limit which host paths Plex can read:
- Put your libraries under **`/media/` or `/mnt/`** and run:
  ```bash
  sudo snap connect plexmediaserver:removable-media
  ```
  That grants Plex (Snap) access to external drives and those mount points.
- If your media is elsewhere (e.g. `/srv/media`), either:
  - Bind-mount into `/mnt` (or `/media`) so Plex can see it, or
  - Bind into **`/var/snap/plexmediaserver/common/Media`** (Snap can always see inside its own common dir).
- In Docker, **bind-mount the exact same host paths** the Plex Snap is using. Sonarr/Radarr must write to the same path Plex scans. If the absolute paths differ, use Docker bind mounts to normalize them.

### 4) Permissions / user IDs
- The Plex Snap runs confined and may use a service account; make sure the media directories are readable (and writable if needed) by that account. Easiest: ensure group/world read on media, or set ACLs:
  ```bash
  sudo setfacl -R -m u:snap_daemon:rx /mnt/media
  ```
  (If `snap_daemon` isn’t present, grant the directory group/world read—or use your preferred ACL policy.)
- Match Docker containers’ `PUID/PGID` to the owner/group of the media so moves/renames from Sonarr/Radarr don’t bounce on permissions.

### 5) Hardware transcoding (optional)
If you use Intel Quick Sync / GPU:
- Make sure the device nodes (e.g. `/dev/dri`) are accessible to Plex. With Snap, you may need to grant the relevant interface and ensure group membership/ACLs on `/dev/dri`. (And for Docker containers doing their own transcoding, pass the device with `--device=/dev/dri`.)

### 6) Webhooks/notifications
- Sonarr/Radarr → Plex notifications (library refresh) work the same: point them to `http(s)://<host>:32400` with your Plex token. Snap vs `.deb` doesn’t matter here.

---

#### Practical checklist
1. Open firewall: `sudo ufw allow 32400/tcp` (if using UFW).
2. Connect removable media:  
   `sudo snap connect plexmediaserver:removable-media`
3. Choose a **single canonical host path** for media (e.g. `/mnt/media`) and:
   - Point Plex libraries there.
   - Bind-mount the same path into Docker containers for Sonarr/Radarr.
4. Fix perms (read for Plex, write for downloaders): use matching `PUID/PGID` or ACLs.
5. Configure Sonarr/Radarr to notify Plex (API URL + token).
